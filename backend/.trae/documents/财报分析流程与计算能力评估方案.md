# 财报分析流程与计算能力评估方案

## Summary

目标：评估“先提取 PDF 数据，再用财报分析 skill 分析”这条链路是否可行，并回答是否需要增加计算工具/计算 skill。

结论（先给结论）：

1. 当前链路**直接不可用**于财报分析（资产负债表），因为现有 `pdf` skill 产物不是财务数值表。
2. 需要补“数值化抽取 + 指标计算”能力；至少要有计算模块，建议落地为可复用计算能力。
3. 计算能力可以先以内嵌模块实现；若后续复用场景增多，再升级为独立 `calc` skill。

## Current State Analysis

### 1) 现有 PDF skill 的真实能力边界

- 当前自动路由到 PDF 时执行脚本：`scripts/extract_form_structure.py`。
- 该脚本输出内容为：
  - `pages/labels/lines/checkboxes/row_boundaries`
  - 侧重“表单结构识别”，非“财报表格数值抽取”。
- 现有解析预览也只统计数量，不含“资产负债项目金额”。

=> 结论：当前 PDF 结果无法直接喂给“财报规则分析”。

### 2) 财报分析需求的数据特征（来自 `taskmd/task.md`）

- 需要对多个科目进行比对与增长分析：货币资金、短期负债、应收、存货、固定资产、在建工程、商誉、长期应收款等。
- 需要指标计算（比率/增速/占比/差异）与规则触发。
- 需要至少两期数据（同比/趋势）才能支撑部分规则。

### 3) 当前系统架构对新增能力的兼容性

- skill 加载机制可无缝新增目录（`SKILL.md` + 脚本）。
- sandbox 可执行 Python 脚本并回传 `stdout/stderr/exitCode`。
- chat 流程支持 skill 执行结果注入回答上下文。
- 触发侧当前偏文档类自动路由，业务分析 skill 主要依赖 LLM 路由。

## Proposed Changes

说明：本阶段仅评估与方案，不执行代码。

### A. 推荐的最小可行流程（MVP）

流程建议拆为 3 段：

1. `pdf_extract_financial_table`（新增或替换 PDF 脚本）
   - 从 PDF 提取资产负债表结构化数据（行名、列名、金额、期次）。
   - 输出统一 JSON（如 `balance_sheet_rows`）。
2. `financial_normalizer`（可与分析脚本合并）
   - 科目别名映射（例如“货币资金/现金及存款”等）。
   - 金额清洗（千分位、单位万元/亿元、负号、空值）。
3. `financial_statement_analysis`（目标 skill）
   - 执行规则 + 指标计算，输出 `signals/risks/evidence/summary`。

### B. 是否要加“计算工具/计算 skill”

推荐答案：**要加计算能力**，但分阶段。

- 阶段 1（推荐）：先加“计算模块（library）”，放在财报 skill 脚本内部调用。
  - 优点：实现快、耦合低、交付风险小。
  - 适合当前只做资产负债表分析。
- 阶段 2（可选）：当出现多业务复用（估值、同业对比、现金流联动）时，再抽成独立 `calc` skill。
  - 优点：复用高、可单独测试与演进。
  - 代价：多一次 skill 编排，调用复杂度提升。

结论：当前阶段**不强制新增独立 calc skill**，但**必须具备可测试的计算模块**。

### C. 文件级落地建议（执行阶段）

- 新增：`backend/.agent/skills/financial_statement_analysis/SKILL.md`
- 新增：`backend/.agent/skills/financial_statement_analysis/scripts/analyze_balance_sheet.py`
- 新增：`backend/.agent/skills/financial_statement_analysis/scripts/compute_metrics.py`
- 新增：`backend/.agent/skills/financial_statement_analysis/scripts/schema.example.json`
- 可选新增：`backend/.agent/skills/pdf/scripts/extract_financial_tables.py`

触发策略建议：

- 第一版通过 LLM 路由触发 `financial_statement_analysis`。
- 后续可在 `chat-service` 增加关键词兜底（“资产负债表分析/财报分析/偿债能力分析”）。

### D. 输出协议建议（保证可解释）

- 分析输出 JSON 最少包含：
  - `input_quality`（字段完备度）
  - `metrics`（如流动比率、现金短债比、应收占比等）
  - `signals`（命中规则列表）
  - `risks`（分级 + 证据）
  - `summary`（给用户的自然语言结论）

## Assumptions & Decisions

- 决策 1：先以“资产负债表分析”作为单一范围，不扩展利润表/现金流量表。
- 决策 2：当前阶段优先做“计算模块”而非独立 calc skill。
- 决策 3：若后续出现多场景复用，再演进为独立 calc skill。
- 假设 1：输入源可提供至少一份可解析资产负债表（PDF/JSON）。
- 假设 2：可接受先交付规则引擎版，不引入复杂机器学习评分。

## 风险评估（问题点）

1. **高风险：PDF 财报版式差异大**
   - 可能抽取错列、错行，导致后续计算错误。
   - 对策：增加抽取置信度与人工校验字段。
2. **高风险：单位/口径不统一**
   - 万元/亿元、合并口径/母公司口径混用会造成误判。
   - 对策：schema 显式单位与口径，缺失则降级为“无法判定”。
3. **中风险：规则阈值行业差异**
   - 固定阈值可能不适配银行、地产、制造等行业。
   - 对策：支持行业参数或阈值配置输入。

## Verification Steps

执行阶段建议验收：

1. 用样例 PDF 抽取后得到结构化数值 JSON（非坐标结构）。
2. 运行财报分析脚本输出完整 `metrics/signals/risks`。
3. Chat 请求“分析资产负债表”可触发 skill 并返回分析结论。
4. 构造缺字段输入时，返回结构化错误而非模型臆测。
5. 回归确认：pdf/docx/xlsx 原有流程不被破坏。
